## 一、项目背景与目标

### 1.1 背景

Linux/嵌入式驱动工程师需频繁编写API接口文档，工作具有以下特点：
- 重复性高：相似的结构体、函数声明需要逐个文档化
- 易出错：手动编写容易出现遗漏或与代码不一致
- 维护难：代码更新后文档同步困难

### 1.2 目标

| 目标项 | 说明 |
|--------|------|
| **输入** | `.c` / `.h` 驱动源码文件 |
| **输出** | 规范的 `.md` API接口文档 |
| **自校验** | 确保文档与代码一致性 |
| **可扩展** | 支持后续扩展至设计文档生成 |

## 三、模块详细设计

### 3.1 输入层 (Input Layer)

| 功能 | 说明 |
|------|------|
| 文件读取 | 支持 `.c` / `.h` 文件 |

### 3.2 解析层 (Parser Layer)

提取以下关键信息：

```
源码解析树
├── 宏定义 (#define)
│   ├── 硬件配置宏 (GPIO, UART, 中断号等)
│   ├── 协议常量 (命令ID, 超时值等)
│   ├── 状态码/错误码
│   └── 宏函数 (如 #define MAX(a,b)) ◀ 新增
│
├── 条件编译处理 (#ifdef/#if/#endif) ◀ 新增：高优先级
│   ├── 解析各分支内容（可配置）
│   └── 记录条件表达式
│
├── 类型定义
│   ├── 枚举 (enum) ── 状态机/模式/结果码
│   ├── 结构体 (struct) ── 参数/结果/上下文
│   ├── 联合体 (union) ◀ 新增：协议数据包常用
│   ├── typedef 定义 ◀ 新增：高优先级
│   │   ├── typedef enum/struct
│   │   └── typedef 函数指针
│   └── 函数指针类型 ◀ 新增：回调函数定义常见
│
├── 函数声明
│   ├── 函数名 + 返回值类型
│   ├── 参数列表 + 类型
│   ├── 参数方向标注 (IN/OUT/INOUT) ◀ 新增
│   ├── 注释描述 (Doxygen风格解析)
│   ├── 函数分类 (初始化/控制/查询/回调)
│   └── 属性标记 (static/deprecated等) ◀ 新增
│
└── 依赖关系
    ├── 头文件包含
    └── 外部依赖
```

### 3.4 校验层 (Validator Layer) - **核心自校验逻辑**

| 校验项 | 检查内容 | 错误级别 | 示例 |
|--------|----------|----------|------|
| **声明-定义一致性** | .h声明 与 .c定义的参数/返回值是否匹配 | ERROR | 函数签名不一致 |
| **结构体完整性** | 结构体字段是否都有注释说明 | WARNING | 字段无描述 |
| **函数覆盖率** | 所有公开函数是否都有文档 | WARNING | 缺少函数文档 |
| **类型引用** | 参数/返回值类型是否已定义 | ERROR | 引用未定义类型 |
| **宏值有效性** | 宏定义值是否合理 | INFO | 超时值为0 |
| **命名规范** | 是否遵循驱动命名规范 | INFO | 无模块前缀 |
| **交叉引用** | 文档中引用的函数/类型是否存在 | ERROR | 引用不存在的函数 |
| **static函数处理** | 内部函数是否需要生成文档（可配置）| INFO | static函数 |
| **deprecated标记检测** | 检测废弃标记属性或注释中的废弃说明 | INFO | 标记废弃函数 |
| **参数方向标注** | IN/OUT/INOUT是否与注释一致 | WARNING | 方向标注不一致 |
| **函数分类一致性** | 函数名与声明的分类是否匹配 | INFO | 命名与分类不符 |
| **线程安全标记** | 函数是否标记了线程安全性（可选）| INFO | 无线程安全说明 |

**校验报告示例：**

```
=== 校验报告 ===
[ERROR] fs5708_face_enroll_single: .h声明参数为param, .c定义参数为p_param
[WARNING] fs5708_ctx_t: 字段 'tx_buf' 缺少注释说明
[WARNING] fs5708_face_verify: 参数 'result' 标注为IN但实际应为OUT
[INFO] FS5708_RESET_TIMEOUT_MS: 建议添加单位说明
[INFO] fs5708_internal_helper: static函数，已配置不生成文档

总计: 1 错误, 2 警告, 2 提示
```

### 4.3 实际工程问题处理 ◀ 新增

#### 4.3.1 增量更新机制

**问题**: 代码变更后如何只更新变化部分，保留手动编辑的内容？

**解决方案**:

```
增量更新流程
├── 1. 解析新旧两版代码，生成IR对比
│   ├── 识别新增/删除/修改的函数
│   └── 识别变更的结构体/枚举
│
├── 2. 文档区域标记
│   ├── <!-- AUTO-GENERATED-START --> ... <!-- AUTO-GENERATED-END -->
│   │   └── 自动生成区域：每次覆盖更新
│   └── <!-- MANUAL-EDIT-START --> ... <!-- MANUAL-EDIT-END -->
│       └── 手动编辑区域：保留用户修改
│
└── 3. 合并策略
    ├── 自动区域：直接替换
    ├── 手动区域：保留不变，输出变更提示
    └── 新增内容：追加到对应章节
```

#### 4.3.2 多模块文档合并

**问题**: 一个驱动可能有多个 `.c`/`.h` 文件，如何合并成一份文档？

**解决方案**:

```yaml
# config/modules.yaml
modules:
  - name: FS5708
    input:
      headers:
        - face_palm_fs5708.h
        - fs5708_protocol.h
      sources:
        - face_palm_fs5708.c
        - fs5708_uart.c
    output:
      file: FS5708_API_Document.md
      merge_strategy: by_category  # by_category / by_file
```

**合并策略**:
- `by_category`: 按接口分类合并（初始化/业务/查询）
- `by_file`: 按源文件分章节

#### 4.3.3 代码版本关联

**问题**: 文档应记录对应代码的版本信息，便于追溯。

**解决方案**:

```
文档头部自动生成版本信息：
---
代码版本: commit:abc123def (2026-03-27)
生成时间: 2026-03-27 14:30:00
Agent版本: v1.0.0
---

变更日志章节自动更新：
| API版本 | 代码版本 | 变更内容 |
|---------|----------|----------|
| v1.1.0 | abc123 | 新增掌静脉校验接口 |
| v1.0.0 | def456 | 初始版本 |
```

#### 4.3.4 Static函数处理策略

**配置项**:

```yaml
# config/validation_rules.yaml
validation:
  static_functions:
    mode: "skip"  # skip / include / auto
    # skip: 不为static函数生成文档
    # include: 全部生成
    # auto: 根据是否有公开注释决定
```

|------|------|------|
| **开发语言** | Python 3.10+ | 丰富的生态，易于维护 |
| **代码解析** | Tree-sitter / pycparser | 准确解析C语法树 |
| **注释提取** | 正则 + LLM辅助 | Doxygen风格优先 |
| **文档生成** | Jinja2模板 | 可定制输出格式 |
| **LLM集成** | Claude API | 智能描述生成/语义理解 |
| **校验引擎** | 规则引擎（自研） | 可配置校验规则 |
| **配置管理** | YAML | 人类可读的配置格式 |

### 6.2 方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **Tree-sitter** | 快速、准确、多语言支持 | 需要编译语法库 | ⭐⭐⭐⭐⭐ |
| **pycparser** | 纯Python、易集成 | 对GNU扩展支持有限 | ⭐⭐⭐⭐ |
| **正则解析** | 简单、无依赖 | 准确性低、难以处理复杂情况 | ⭐⭐ |
| **LLM直接解析** | 智能、灵活 | 成本高、不稳定 | ⭐⭐⭐ |

**推荐方案**: Tree-sitter (主要) + LLM辅助 (注释理解/描述生成)
